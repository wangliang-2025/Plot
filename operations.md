# Plot 项目运维管理手册（长期维护版）

本文档定义项目上线后的标准运维流程，目标是做到：可观测、可恢复、可追溯、可持续迭代。

---

## 1. 运维目标与 SLO

建议先定义可执行的服务目标（可按团队情况调整）：

- 可用性：`>= 99.9%`
- 关键页面错误率（5xx）：`< 1%`
- API P95 响应时间：`< 800ms`
- 故障响应：`5 分钟内确认，30 分钟内止损`

---

## 2. 角色与职责

- **Owner（系统负责人）**：发布决策、故障总协调、权限审批
- **开发负责人**：代码修复、迁移评估、回滚实施
- **运维负责人**：监控告警、平台配置、备份恢复

小团队可由 1 人兼任，但职责应在文档中保留，便于后续扩展。

---

## 3. 监控与告警基线

## 3.1 必备监控项

### Vercel

- 部署成功率
- Serverless Function 错误率
- 冷启动/超时趋势

### Neon

- 活跃连接数
- CPU、I/O、查询耗时
- 长事务与异常会话

### 应用侧

- 登录成功率
- 文章读取/写入成功率
- 后台关键接口错误率

## 3.2 告警分级

- **P1（立即处理）**：站点不可访问、数据库不可用、登录全面失败
- **P2（尽快处理）**：错误率上升、核心接口超时明显增加
- **P3（排期处理）**：性能轻微下降、非核心功能异常

---

## 4. 日常巡检（日/双日）

每 1~2 天执行一次：

1. 检查最新部署状态（是否失败、是否回滚）
2. 冒烟测试（首页、登录、列表、详情、写入）
3. 查看 Vercel Function 错误日志
4. 查看 Neon 连接与负载曲线
5. 记录异常并创建待办

建议维护 `ops-log`，至少记录：

- 时间
- 操作人
- 异常现象
- 处理结果
- 后续动作

---

## 5. 周维护（每周固定时段）

1. 依赖安全检查（漏洞告警）
2. 小版本升级评估（先测试后上线）
3. 备份可用性验证（抽样恢复）
4. 容量趋势分析（流量、缓存、数据库）
5. 复盘本周异常并补流程

---

## 6. 月维护（治理层）

1. 权限审计（谁可改生产变量、谁可发版）
2. 密钥轮换（数据库密码、第三方 token）
3. 成本审计（Vercel 调用、Neon 资源）
4. 数据库结构治理（索引、慢查询、冗余数据）
5. 文档审计（部署文档与现状是否一致）

---

## 7. 备份与恢复管理

## 7.1 备份策略

- 生产数据库依赖 Neon 官方快照
- 关键变更前执行一次手工快照（标记版本）
- 建议额外导出逻辑备份（如 `pg_dump`）用于异地灾备

## 7.2 恢复演练

至少每月一次：

1. 在非生产环境恢复备份
2. 验证核心表与关键业务流程
3. 记录恢复耗时与问题

目标：

- RTO（恢复时间）有实测数据
- RPO（数据可接受损失）有明确阈值

---

## 8. 变更与发布管理

## 8.1 发布前检查

- [ ] PR 已评审
- [ ] 风险点已说明
- [ ] `prisma migrate status` 正常
- [ ] 迁移执行计划明确（仅执行一次）
- [ ] 回滚方案已准备

## 8.2 发布步骤（生产）

1. 合并到 `main`
2. 执行迁移（串行单次）
3. 执行构建与部署
4. 发布后冒烟测试
5. 观察指标 15~30 分钟

## 8.3 回滚策略

- 优先回滚应用版本（最快恢复可用性）
- 数据库迁移不建议盲目回滚，优先 forward-fix
- 极端场景再走备份恢复

---

## 9. Prisma + Neon 专项运维（重点）

### 9.1 连接规范

- `DATABASE_URL`：应用运行（pooler）
- `DIRECT_URL`：迁移执行（direct）
- `schema.prisma` 必须配置 `directUrl`

### 9.2 迁移规范

- 迁移只在一个流程里执行
- 不要在 `postinstall`、`build`、`deploy hook` 多处重复迁移
- 避免并行部署同时触发 `prisma migrate deploy`

### 9.3 P1002 标准处置

症状：

- `Timed out trying to acquire a postgres advisory lock`

处置顺序：

1. 确认是否并发部署
2. 检查 `DIRECT_URL` 是否正确
3. 重试迁移（排除瞬时抖动）
4. 查看 Neon 活跃连接与长事务
5. 修正部署流程，确保迁移串行

---

## 10. 故障应急 Runbook

## 10.1 应急流程

1. 发现告警（自动/人工）
2. 5 分钟内确认故障等级（P1/P2/P3）
3. 立即止损（回滚、降级、限流）
4. 定位根因（日志、指标、最近变更）
5. 输出复盘报告

## 10.2 复盘模板

- 事故时间线
- 影响范围与持续时间
- 根因
- 修复措施
- 预防措施（告警、流程、代码）

## 11. 常用运维命令

### 11.1 本地/CI 核心命令

```bash
npx prisma validate
npx prisma generate
npx prisma migrate status
npx prisma migrate deploy
npm run build
```

### 11.2 Git 运维操作

```bash
git status
git pull origin main
git checkout -b hotfix/issue-name
git add .
git commit -m "fix: resolve production issue"
git push -u origin hotfix/issue-name
```

### 11.3 排障辅助

```bash
# 查看当前环境变量是否注入（注意不要在公开日志打印敏感值）
echo $DATABASE_URL
echo $DIRECT_URL
```

---

## 12. 权限与安全管理

- 最小权限原则（生产变量仅少数人可编辑）
- 定期清理无效账号与 token
- 严禁将 `.env`、数据库凭据提交到仓库
- 对管理员账号启用强密码策略

## 14. 维护执行节奏建议（可直接照搬）

- 每天：5~10 分钟巡检
- 每周：30~60 分钟维护
- 每月：1 次治理评审（权限、成本、性能、恢复）
- 每季度：一次完整应急演练（含恢复）

长期稳定的关键，不是“零故障”，而是“出问题可快速感知、快速止损、快速修复”。
